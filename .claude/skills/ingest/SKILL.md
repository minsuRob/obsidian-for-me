# /ingest — Inbox → Wiki 정제 파이프라인

카파시 LLM Wiki 원칙: **원본은 절대 수정하지 않고**(플래그만 추가), AI가 읽어 wiki를 유기적으로 성장시킨다. wiki는 계속 자란다.

기준 경로: `Work/Markhub-LLMWiki/` (이하 `WIKI/`).

## 처리 순서

### 1. 미처리 원본 수집
- `WIKI/inbox/*.md` 중 frontmatter에 `Processed: true`가 **없는** 파일만 대상.
- 대상이 없으면 "정리할 새 항목이 없습니다"라고 알리고 종료.
- 여러 개면 오래된 날짜순으로 하나씩 처리.

### 2. 원본 분해 (섞인 정보 쪼개기)
각 원본을 읽고 정보를 종류별로 분류한다. 예:
- 일정/마일스톤 변경 → 해당 `Feature-*` 문서의 `## 진행 상황`
- 벤더/계약/수수료/단가 → 해당 `Vendor-*` 문서의 표
- 결정 사항 → 해당 문서 `## 결정 사항`
- 새 이슈/버그/할 일 → 해당 `Feature-*`의 `## 열린 이슈` 표
- 회의 자체 요약 → `WIKI/wiki/meetings/<날짜>_<주제>.md` 신규 생성

### 3. 대상 wiki 문서 찾기 / 만들기
- 먼저 `WIKI/wiki/**` 를 훑어 관련 문서를 찾는다(제목·태그·내용으로 매칭).
- 있으면 **해당 섹션만 정밀 수정**한다. 문서 전체를 갈아엎지 말 것.
- 없으면 알맞은 폴더에 새로 만든다:
  - `wiki/features/Feature-<이름>.md`, `wiki/vendors/Vendor-<이름>.md`
  - frontmatter 컨벤션 준수(아래) 및 관련 문서를 `[[링크]]`로 연결.

### 4. 반영 규칙 (구체 예시 = 토스 회의록)
- `Feature-Payment.md` → `## 진행 상황`의 오픈 목표를 원본대로 갱신(예: `7월 말` → `8월 중순`). 값을 바꾸되 근거를 `## 히스토리`에 1줄 남긴다.
- `Vendor-TossPay.md` → 계약/수수료 표의 수수료율 행을 새 값(예: `1.8%`)으로 수정하고 기준일을 오늘로.
- 변경한 문서의 frontmatter `최종수정`을 오늘 날짜로 갱신.
- 새 결정/이슈는 관련 문서의 표·목록에 **추가**(기존 항목 삭제 금지).
- 문서 간 관계가 생기면 양쪽에 `[[링크]]`를 건다(백링크 형성).

### 5. 회의 요약본 생성
- 회의성 원본이면 `wiki/meetings/<YYYY-MM-DD>_<주제-케밥>.md`에 정제 요약 작성:
  핵심 요약 / 결정 사항 / 액션 아이템, 관련 문서 `[[링크]]` 포함. frontmatter `유형: meeting`.

### 6. 원본 아카이브 (중복 처리 방지)
- 원본 파일 frontmatter 맨 위에 `Processed: true` 와 `처리일: <오늘>` 추가. **본문은 건드리지 않는다.**
- 파일을 `WIKI/inbox/` → `WIKI/raw/` 로 이동(`git mv` 대신 일반 이동, 필요시 mkdir).

### 7. 요약 보고
처리 결과를 사용자에게 표로 보고:
- 처리한 원본, 수정된 wiki 문서 + 무엇이 바뀌었는지, 새로 만든 문서, 이동된 원본.

## Frontmatter 컨벤션 (한글 키)
```yaml
유형: feature | task | vendor | meeting
상태: 대기 | 진행중 | 완료 | 보류
우선순위: 높음 | 중간 | 낮음
담당: 나
관련: ["[[관련문서]]"]
최종수정: YYYY-MM-DD
태그: []
```

## 주의
- 원본(raw) 내용은 절대 수정하지 않는다(플래그/이동만).
- 확실치 않은 정보는 지어내지 말고, 요약 보고에 "확인 필요"로 표시.
- 파괴적 편집(문서 전체 재작성) 금지. 항상 섹션 단위 최소 수정.
