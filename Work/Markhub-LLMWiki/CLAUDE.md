# Markhub LLM Wiki — 스키마

> 이 문서는 **AI(Claude)에게 이 위키를 어떻게 관리해야 하는지 알려주는 규칙서**입니다.
> 카파시의 LLM Wiki 패턴( https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f )을
> Markhub 업무용으로 옮긴 것. AI는 이 위키에서 작업할 때 **항상 이 파일을 먼저 읽고 규칙을 따릅니다.**

---

## 이 위키가 하는 일 (한 줄 요약)

거칠게 던져둔 메모·회의록·일감을, AI가 읽고 정제해서 **서로 연결된 위키 페이지로 축적**합니다.
그래서 "지난주에 뭐 했지?", "그 이슈 어떻게 됐지?"를 매번 뒤지지 않고 **즉시 답**을 얻습니다.

---

## 3층 구조 (가장 중요한 규칙)

```
Work/Markhub-LLMWiki/
├── CLAUDE.md        ← 규칙서 (지금 이 파일)
├── INDEX.md         ← 위키 전체 목차. 질문 받으면 여기부터 읽는다
├── LOG.md           ← 무슨 작업을 언제 했는지 기록 (append-only)
├── raw/             ← 원본층. 사람이 던지는 곳. AI는 읽기만, 절대 수정 금지 ❗
│   ├── README.md    ← 사용 안내
│   └── daily/       ← 날짜별 데일리노트 (YYYY-MM-DD.md)
├── wiki/            ← 정제층. AI가 전적으로 소유·생성·갱신
│   └── tasks/       ← 일감 페이지 (업무 트래킹의 핵심)
└── templates/       ← 데일리노트 템플릿
```

**세 가지 층의 역할이 절대 섞이면 안 됩니다:**

| 층 | 누가 쓰나 | AI의 권한 |
|----|----------|----------|
| **raw/** | 사람이 아무렇게나 던짐 | **읽기 전용. 수정·삭제 금지** |
| **wiki/** | AI가 만들고 유지 | 전적으로 소유. 생성/갱신/링크 |
| **CLAUDE.md / INDEX.md / LOG.md** | AI가 관리 | 규칙 따라 갱신 |

> 원본을 절대 건드리지 않는 것이 이 패턴의 핵심입니다. 원본은 그대로 두고 wiki만 계속 자랍니다.

## 위키 페이지 작성 관례

모든 wiki 페이지는 아래 형식을 따릅니다. **한국어**로 작성합니다.

### 프론트매터 (YAML)

```yaml
---
유형: 일감          # 일감 | 회의 | 개념 | 사람
상태: 진행중        # 진행중 | 완료 | 보류 | 대기   (일감/회의에만)
우선순위: 최우선     # 최우선 | 중간 | 하순위        (일감에만)
생성일: 2026-07-02
갱신일: 2026-07-02
출처:               # 이 페이지가 근거로 삼은 raw 파일들
  - "[[핵심요소.]]"
---
```

### 본문 구조 (일감 페이지 기준)

```markdown
# {일감 제목}

## 배경
왜 이 일이 필요한가, 어디서 나왔는가.

## 현황
지금 어디까지 됐는가. (여기가 일일보고·주간회의의 재료)

## 타임라인
- 2026-07-02: 이슈 최초 확인 (출처: [[핵심요소.]])

## 관련
- [[다른 일감 페이지]]
- [[관련 회의]]
```

### 링크 규칙
- 페이지 사이 연결은 반드시 `[[페이지 제목]]` 형식 (옵시디언 그래프·역링크 작동).
- 파일명 = 페이지 제목. 링크가 실제 파일명과 정확히 일치해야 함.
- 관련 페이지끼리 **양방향으로** 링크를 건다 (A가 B를 링크하면 B에도 A를 링크).

## AI 워크플로우 (요청별 행동 규칙)

### 1. Ingest — "이거 정리해줘" / `/ingest`
raw 문서를 위키로 흡수하는 작업. 순서:
1. 대상 raw 파일을 읽는다 (원본은 수정하지 않음).
2. 핵심 발견을 사용자에게 한두 줄로 짚어준다.
3. 관련 wiki 페이지를 **생성하거나 갱신**한다 (한 번에 여러 페이지 건드릴 수 있음).
4. 페이지 간 `[[링크]]`를 양방향으로 연결한다.
5. `INDEX.md`에 새 페이지를 등록/갱신한다.
6. `LOG.md`에 기록을 남긴다.

### 2. Query — "그거 어떻게 됐지?" (질문)
1. **먼저 `INDEX.md`를 읽어** 관련 페이지를 찾는다 (전체 파일을 뒤지지 않는다).
2. 해당 페이지들을 읽고 종합해 답한다. **출처(raw)를 함께 인용**한다.
3. 답하는 과정에서 알게 된 가치 있는 내용은 **위키에 다시 저장**한다 (채팅에서 휘발시키지 않음).

### 3. 일일보고 — "오늘 일일보고 만들어줘"
1. 오늘 날짜의 `raw/daily/YYYY-MM-DD.md`를 읽는다 (없으면 사용자에게 오늘 한 일을 묻거나, 오늘 갱신된 일감 페이지를 근거로 삼는다).
2. 오늘 언급·갱신된 일감 페이지들의 **현황**을 종합한다.
3. 아래 형식으로 **초안**을 데일리노트의 `## 일일보고` 섹션에 작성한다:
   ```
   ### 오늘 한 일
   - …
   ### 이슈 / 블로커
   - …
   ### 내일 할 일
   - …
   ```
4. 진행된 내용을 해당 **일감 페이지의 현황·타임라인에 반영**하고 `갱신일`을 오늘로 바꾼다.
5. `LOG.md`에 기록한다.

### 4. 주간회의 — "주간회의 자료 만들어줘"
1. 이번 주 `raw/daily/` 노트들 + `LOG.md` + 일감 페이지 상태를 모은다.
2. 아래 안건 **초안**을 만든다:
   ```
   ## 이번 주 주간회의 (YYYY-MM-DD ~ YYYY-MM-DD)
   ### ✅ 완료
   ### 🔄 진행중
   ### 🚧 블로커 (논의 필요)
   ### 📋 다음 주 계획
   ```
3. 각 항목에 근거 일감 페이지를 `[[링크]]`로 단다.
4. `LOG.md`에 기록한다.

### 5. Lint — "점검해줘" / `/lint`
위키가 썩지 않게 정기 점검. 아래를 찾아 리포트하고 **사용자 확인 후** 수정:
- INDEX.md에 빠진 페이지 / 존재하지 않는 페이지
- 아무도 링크하지 않는 고아 페이지
- 깨진 `[[링크]]` (대상 파일 없음)
- 30일 이상 갱신 안 된 '진행중' 일감
- 서로 모순되는 서술

## LOG.md 기록 형식

항상 아래 한 줄 형식으로 **맨 아래에 추가** (append-only, 위를 지우지 않음):

```
## [YYYY-MM-DD] {ingest|query|daily|weekly|lint} | {제목/요약}
- 무엇을 했는지 한두 줄
```

## 톤 & 원칙
- 사람은 **통찰과 판단**에 집중, AI가 **정리·연결·일관성 유지**의 잡일을 맡는다.
- 확실하지 않으면 지어내지 말고 raw 출처를 확인하거나 사용자에게 묻는다.
- 페이지는 사람이 읽기 좋게, 동시에 다음번 AI가 처리하기 좋게 작성한다.
